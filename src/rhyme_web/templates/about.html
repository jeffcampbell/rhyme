About Rhyme

What is this?

Rhyme is a tool for evaluating how well AI models correlate microservice incidents by proximal cause — not just by symptoms. It answers the question: "When your AI SRE tool surfaces a similar past incident, is it actually helpful, or is it a red herring?"

A red herring is an incident that looks similar (same error codes, same affected services, same symptoms) but has a completely different underlying cause. Surfacing red herrings is actively harmful — the on-call engineer wastes time pursuing the wrong fix.

How it works

  1. Import your incidents — paste or upload a JSON file with your organization's real incident data. Minimum fields: id, summary, timestamp. No logs or alerts needed.
  2. Pairs are generated — the tool creates incident pairs for you to evaluate. If you've connected a model, pairs are sampled across the model's confidence spectrum (high/medium/low/random) to calibrate both true positives and false positives. Without a model, pairs are sampled randomly.
  3. SREs label pairs — for each pair, an SRE answers: "Would knowing about Incident A help you respond to Incident B?" with Yes, No, or Maybe. This takes about 10 seconds per pair.
  4. Results appear on the dashboard — the tool scores model predictions against your team's labels, showing precision, recall, calibration, and red herring detection.

Methodology

Pair sampling

If model correlation scores are available, pairs are stratified across four buckets:

  • High confidence (25%) — pairs the model thinks are related. Verifies true positives.
  • Medium confidence (25%) — the interesting boundary where the model is uncertain.
  • Low confidence (25%) — pairs the model thinks are unrelated. Verifies true negatives.
  • Random (25%) — unbiased calibration anchors, independent of the model's opinion.

Without model scores, all pairs are sampled randomly.

Labeling

The question "Would knowing about Incident A help you respond to Incident B?" is intentionally broader than "Do these have the same root cause?" because in practice, SREs benefit from seeing related incidents even when the proximal cause differs slightly. The three-option scale:

  • Yes — knowing about this incident would meaningfully help respond to the other one.
  • Maybe — there's some relevance but it's not clear-cut. Excluded from precision/recall calculations.
  • No — these incidents are unrelated. Surfacing one during the other would be a distraction.

Scoring

The dashboard reports:

  • Precision & Recall at multiple thresholds (0.5, 0.7, 0.9) — how well the model's confidence scores predict human agreement.
  • Calibration — is the model overconfident? Bucketed by confidence range, showing the gap between what the model claims and what humans actually agree with.
  • Red herring detection — counts of cases where the model is confident (>0.7) but humans said "no." These are the dangerous false positives.
  • Inter-annotator agreement — Cohen's kappa between labelers, if multiple SREs label the same pairs. Low agreement suggests the task is ambiguous or labelers need calibration.

Statistical notes

We recommend labeling at least 100 pairs for statistically meaningful results, ideally 200. With 2+ labelers on overlapping pairs, inter-annotator agreement becomes available. "Maybe" labels are excluded from precision/recall but counted in the label distribution.

FAQ

How many incidents do I need?

Minimum 5, but you'll get more useful results with 50+. With 50 incidents there are 1,225 possible pairs — the tool samples up to 200 for labeling.

How long does labeling take?

About 10 seconds per pair for an experienced SRE. 200 pairs takes roughly 30-40 minutes. You can stop and resume at any time — progress is saved after every label.

What format should my incident data be in?

JSON, either a bare array or an object with an "incidents" key. Each incident needs:

  • id — unique identifier (e.g., "INC-001")
  • summary — human-written description of what happened (any length — long summaries are truncated in the UI with a "show more" toggle)
  • timestamp — ISO 8601 timestamp

Optional fields: severity, service, url (link to your incident tracker — shown as "View in tracker" during labeling), metadata (arbitrary key-value pairs).

Can multiple people label at the same time?

Yes. The tool uses a database backend (PostgreSQL in production) that handles concurrent access safely. Each label records the labeler's ID. When multiple labelers label the same pair, their responses are aggregated by majority vote for scoring, and inter-annotator agreement is computed.

What does the "Maybe" option mean for scoring?

"Maybe" labels are excluded from precision and recall calculations — they don't count as positive or negative. They are tracked separately in the label distribution so you can see how often labelers are uncertain. A high "maybe" rate may indicate that your incidents need more detail in their summaries.

What are "red herrings" in the dashboard?

Red herrings are incident pairs where the model is confident they're related (confidence >0.7) but your SREs said "no." These are the most dangerous failure mode — a model that confidently surfaces unrelated incidents can actively harm incident response by sending the on-call down the wrong path.

How do I connect my model?

Currently, the model's correlation scores can be provided during pair sampling via the API. The tool will then stratify pairs across the model's confidence spectrum. If no model scores are provided, pairs are sampled randomly and the dashboard shows label distribution without model-vs-human comparison.

Is my incident data stored securely?

All data stays on your infrastructure — this tool runs entirely within your environment (Docker container + PostgreSQL database). No data is sent to external services. For production deployments, we recommend running behind your corporate VPN or SSO.

How is this different from the synthetic benchmark?

Rhyme includes two complementary evaluation tools. This web labeling tool is one of them. The other is a synthetic benchmark — see below.

The synthetic benchmark

The synthetic benchmark evaluates models against a controlled corpus of 1,000 generated incidents with known ground-truth labels. It runs from the command line and produces deterministic scores — useful for model selection, prompt engineering, and comparing approaches.

Corpus

1,000 synthetic incidents across 20 proximal cause classes organized into 8 families:

Each class has defined confusable partners — cause classes that share symptoms but require different remediation. These pairs are assigned difficulty tiers (easy, medium, hard) and scored separately, so strong performance on easy cases can't mask failure on the hard ones.

Incident prose is generated using LLM-produced text pools styled after real AWS, Azure, and GCP status page reports. An adversarial style probe verifies that a classifier cannot predict cause class from writing style alone — ensuring models must reason about the incident content, not exploit stylistic patterns.

Three evaluation tasks

TaskWhat it measuresToken cost
Task 1: Full-corpus retrieval Model sees all 1,000 incidents and must rank the most cause-similar ones. Tests both retrieval and reasoning. ~75,000 tokens/query
Task 2: Reasoning-only A BM25 keyword retriever pre-filters to the top 20 candidates. The model re-ranks from this smaller set. Isolates reasoning quality from retrieval quality. ~2,000 tokens/query
Task 3: Remediation Given an incident and 5 multiple-choice remediation options, the model picks the best fix. Options include the correct fix, a symptom-masking fix, a harmful fix, and distractors. ~500 tokens/query

Metrics

All core metrics are deterministic — no LLM-judge in the scoring.

  • Retrieval@k — does the top-k contain any correct match? Reported per difficulty tier.
  • Cause-match precision@k — fraction of top-k sharing the same proximal cause, reported as lift over class base rate to normalize for class-frequency imbalance.
  • Confusion matrix — which wrong classes dominate predictions? Shows exactly where a model fails.
  • Calibration ECE — are the model's confidence scores trustworthy? Bucketed by confidence range.
  • Remediation grades — distribution over {fixed, no-op, masks-symptom, would-worsen}. The would-worsen rate is safety-critical and always reported separately.
  • Efficiency — tokens/query, tokens/correct-match, precision-per-1k-tokens. Enables quality-vs-cost comparison across prompting strategies.

How to use this alongside the web tool

Use the synthetic benchmark for model selection — comparing models, prompting strategies, and approaches in a controlled setting. Use the web labeling tool for validation — confirming the selected model works well on your organization's specific incident patterns.

A model that scores well on the benchmark but poorly on your real incidents likely has a domain mismatch. A model that scores well on your incidents but poorly on the benchmark may be overfitting to your specific vocabulary.
